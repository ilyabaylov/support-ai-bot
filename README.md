# Support-ai-bot

Telegram-бот — AI-администратор для салона красоты. Отвечает клиентам про услуги, цены, запись и акции, понимает текст и голосовые, ведёт живой диалог и, когда нужно, передаёт разговор живому администратору. Управление — кнопками, без команд.

> ⚠️ **Демо-проект.** Салон «Lumière Beauty Studio», адрес, телефон, имена мастеров и цены — вымышленные, приведены для примера. Чтобы запустить под реальный бизнес — поменяйте `knowledge_base/faq.md` и `app/content.py`.

## Зачем
Хотел собрать не «бота по командам», а администратора, с которым клиент общается как с человеком: сам разбирается в запросе, подсказывает цены строго по базе салона и не выдумывает цифры. Если случай сложный — спокойно зовёт живого администратора.

## Что умеет
- **Отвечает по базе знаний (RAG).** Находит нужные куски `faq.md` через эмбеддинги и отвечает, опираясь на них.
- **Держит контекст диалога.** Помнит последние сообщения, понимает уточнения вроде «а в субботу?».
- **Понимает голосовые.** Расшифровывает аудио и отвечает текстом.
- **Стримит ответ.** Печатает по мере генерации, как живой человек.
- **Передаёт администратору**, когда клиент готов записаться, жалуется или прямо просит человека.
- **Админка в Telegram:** статистика, необработанные обращения, пересборка базы — всё кнопками.

## Как отвечает (вкратце)
1. Вопрос клиента → эмбеддинг → поиск ближайших разделов `faq.md`.
2. Найденные куски подмешиваются в промпт как факты о салоне — точные цены, адрес и часы бот берёт только оттуда.
3. Если близких данных в базе мало (score ниже `SIM_THRESHOLD`) — бот не молчит и не отказывает, а отвечает в общих чертах и предлагает уточнить у мастера. Порог здесь — мягкая подсказка модели, а не жёсткий стоп.
4. Ответ стримится клиенту.
5. Если модель решает, что нужен живой администратор, она ставит служебный тег `[[HANDOFF]]` (клиент его не видит) — бот пересылает диалог администратору.

## Стек
- Python 3.12, aiogram 3
- Google Gemini (`google-genai`) — генерация, эмбеддинги, расшифровка голоса
- Локальный RAG на косинусном сходстве, индекс в JSON (без внешних векторных БД)
- SQLite — логи и статистика
- Docker + docker-compose

## Локальный запуск
1. Виртуальное окружение и зависимости:
python -m venv .venv
source .venv/bin/activate        # Windows: .venvScriptsactivate
pip install -r requirements.txt
2. Переменные окружения:
​
cp .env.example .env
   Впишите в `.env` свои `BOT_TOKEN`, `GEMINI_API_KEY` и `ADMIN_IDS`.
3. Соберите индекс (один раз перед стартом и после правок FAQ):
​
python -m scripts.build_index
4. Запустите бота:
​
python -m app.bot
Коротко через Makefile: `make install && make index && make run`.

## Запуск в Docker
​
cp .env.example .env            # и заполните .env
python -m scripts.build_index   # собрать индекс в ./data
docker compose up -d --build

## Переменные окружения (.env)
| Переменная | Что это |
|---|---|
| `BOT_TOKEN` | токен бота от @BotFather |
| `GEMINI_API_KEY` | ключ Google Gemini (https://aistudio.google.com/apikey) |
| `ADMIN_IDS` | Telegram ID админов через запятую |
| `OPERATOR_CHAT_ID` | куда слать обращения (по умолчанию — первому из `ADMIN_IDS`) |
| `SIM_THRESHOLD` | порог уверенности 0..1 (по умолчанию `0.62`) |
| `TOP_K` | сколько кусков базы подмешивать (по умолчанию `4`) |
| `CHAT_MODEL` | модель генерации (`gemini-2.5-flash`) |
| `EMBED_MODEL` | модель эмбеддингов (`gemini-embedding-2`) |

## Структура проекта
​
app/
bot.py            — точка входа и инициализация aiogram
config.py         — конфиг из .env
content.py        — профиль салона и личность администратора (промпт)
dialog.py         — память диалога по пользователям
keyboards.py      — клавиатуры и меню
handlers/         — хендлеры (вопросы, админка, передача оператору)
rag/              — поиск и пересборка индекса
services/         — Gemini и статистика (SQLite)
knowledge_base/faq.md   — база знаний салона
scripts/build_index.py  — пересборка индекса
tests/                  — тесты на RAG-математику и парсинг базы (pytest)

## Ограничения и что дальше
- Индекс и статистика — локальные файлы (JSON + SQLite); для большой базы стоит вынести в нормальную векторную БД.
- Запись пока не интегрирована с CRM/календарём — бот собирает данные и передаёт администратору.
- Бот на long polling, без вебхуков и горизонтального масштабирования.
- Автотесты покрывают RAG-математику и парсинг базы, но не сами хендлеры.
